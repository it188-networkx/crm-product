# Feature Definition - [F-04] Go/No-Go 决策标注与提交

## 特性概述 (Feature Overview)

**所属 Epic**：T-03/E-04 Go/No-Go 验收看板

**业务定义**：
本特性为 CEO 提供对当期策略、渠道或市场的正式决策标注能力。CEO 可在看板中选定决策对象，明确标注"Go（继续执行）"或"No-Go（停止/调整）"，并填写必填的决策依据说明后提交。提交后，系统将该决策写入不可篡改的审计日志，并向相关责任人触发通知。决策一经提交即生效，不可修改；如需更新判断，须以新建独立决策记录的方式进行。

**范围边界**：
1. 包含：CEO 对当期策略/渠道/市场的 Go 或 No-Go 标注操作；决策依据的必填录入；决策提交后的审计日志写入；提交后向相关责任人推送决策通知；已提交决策记录的只读查看
2. 不包含：历史已提交决策的撤回或修改（须以新决策记录替代）；KPI 达成率数据的录入与计算（由 F-01 负责）；跨期策略的多期汇总评级（属 Epic 级流程编排范畴）；通知渠道配置与通知模板管理

## 业务规则 (Business Rules)

- **[BR-F04-01] 决策主体唯一性**：仅 CEO 角色可发起并提交 Go/No-Go 决策标注；其他角色对决策标注操作不具有写权限，只可查看已提交的决策记录（只读）。

- **[BR-F04-02] 决策对象范围**：决策标注仅针对当期（当前考核周期）的策略、渠道或市场维度对象；不可对历史周期的已关闭对象新建决策，[推导-待确认] 也不可对未来周期的对象预先标注。

- **[BR-F04-03] 决策值约束**：每次标注须从"Go（继续执行）"与"No-Go（停止/调整）"二值中选择其一，不可为空，不可选择中间状态。

- **[BR-F04-04] 决策依据必填**：提交时"决策依据"说明文字为必填项，不可为空白。[推导-待确认] 建议最短字符数不低于 10 个字，以保证依据的有效性。

- **[BR-F04-05] 提交不可逆性**：决策一经提交即进入已生效状态，不可撤回、修改或删除。如需更新判断，CEO 须以同一决策对象新建一条独立决策记录，新记录不覆盖旧记录，两者均完整保留在审计日志中。

- **[BR-F04-06] 同期同对象多次决策**：允许 CEO 对同一周期同一决策对象多次提交决策（如先 Go、后 No-Go），每次提交均独立写入审计日志，记录时间戳排序，[推导-待确认] 以最新一条作为当前有效结论展示，旧记录仍完整保留。

- **[BR-F04-07] 决策审计留存**：参见 Epic [BR-04]。所有提交操作须实时写入不可篡改审计日志（GC-04），审计条目包含：操作人、操作时间、决策对象标识、决策值、决策依据全文。

- **[BR-F04-08] 决策记录合规保留期**：参见 Epic [BR-06]。Go/No-Go 决策记录保留期 ≥ 36 个月，系统不提供批量删除或清空决策历史的操作入口。

- **[BR-F04-09] 通知触发时机**：决策成功提交后，系统须在当次提交完成时即时触发通知；通知须包含：决策对象名称、决策结论（Go/No-Go）、决策依据摘要、提交人（CEO）及提交时间。[推导-待确认] 通知失败不阻塞决策提交本身的成功，但须记录通知失败事件以供补发。

- **[BR-F04-10] 权限隔离**：参见 Epic [BR-05]。通知推送对象须严格按角色权限与责任归属过滤，不得将决策内容推送至无权限的角色。

## 业务对象定义 (Business Object Definition)

### 决策标注记录（Go/No-Go Decision Record）

记录 CEO 针对当期某一业务对象（策略/渠道/市场）所作出的正式 Go 或 No-Go 判断，是本特性的核心业务对象。每次提交产生一条独立的不可变记录，多条记录可对应同一决策对象（支持同期多次提交场景）。

| 业务属性 | 业务说明 |
| :--- | :--- |
| 决策记录编号 | 系统自动生成的唯一标识，用于审计追溯与通知关联 |
| 决策周期 | 本条决策所属的考核周期（如 2025-Q2），由系统基于当前期自动填充，[推导-待确认] 不允许手动修改 |
| 决策对象类型 | 决策所指向的业务维度，取值范围：策略 / 渠道 / 市场 |
| 决策对象名称 | 具体被标注的策略名称、渠道名称或市场名称，关联 F-01 的 KPI 数据上下文 |
| 决策结论 | Go（继续执行）或 No-Go（停止/调整），二值必填 |
| 决策依据 | CEO 填写的文字说明，为必填项，解释本次决策的业务理由 |
| 提交人 | 执行提交操作的 CEO 账号标识，由系统自动记录，不可人工填写 |
| 提交时间 | 系统记录的提交操作精确时间戳（含时区），不可人工修改 |
| 记录状态 | 已生效（提交后唯一状态），无草稿态，无可撤销态 |

### 决策通知（Decision Notification）

当 CEO 完成 Go/No-Go 决策提交后，系统向相关责任人推送的通知对象，承载决策结论与关键摘要，驱动责任人及时响应。

| 业务属性 | 业务说明 |
| :--- | :--- |
| 通知编号 | 系统自动生成的唯一标识，用于追踪通知状态 |
| 关联决策记录编号 | 指向触发本通知的决策标注记录 |
| 通知接收人 | 与决策对象相关的责任人账号列表，[推导-待确认] 依据决策对象类型与责任归属关系自动匹配 |
| 通知内容摘要 | 包含决策对象名称、决策结论、决策依据摘要、提交人及提交时间 |
| 通知触发时间 | 决策提交成功后系统发出通知的时间戳 |
| 通知状态 | [推导-待确认] 取值：已发送 / 发送失败；失败须记录以支持补发 |

## 验收标准 (Acceptance Criteria)

| AC | 前置条件 | 触发动作 | 预期结果 | US Ref |
| :--- | :--- | :--- | :--- | :--- |
| AC-01 | CEO 已登录，当期看板存在至少一个可标注的决策对象（策略/渠道/市场） | CEO 选定决策对象，选择"Go"或"No-Go"，填写决策依据，点击"提交" | 系统校验通过，决策标注记录成功写入，页面呈现提交成功反馈，该记录进入只读状态 | US-01 |
| AC-02 | CEO 已选定决策对象并选择了决策结论 | CEO 将决策依据留空，点击"提交" | 系统阻止提交，明确提示"决策依据为必填项"，页面保持可编辑状态等待修正 | US-01 |
| AC-03 | CEO 已登录 | CEO 尝试对历史已关闭周期的对象发起 Go/No-Go 标注 | 系统拒绝操作，提示当前决策仅限本期对象，历史周期对象不可新建决策 | US-01 |
| AC-04 | CEO 成功提交一条决策标注记录 | 提交操作完成后，在审计日志中查询该记录 | 审计日志中存在对应条目，包含：提交人账号、精确提交时间、决策对象标识、决策结论、决策依据全文，且该条目不可被修改或删除 | US-02 |
| AC-05 | CEO 成功提交一条决策标注记录，且存在与决策对象关联的相关责任人 | 提交操作完成后 | 相关责任人收到通知，通知内容包含：决策对象名称、决策结论（Go/No-Go）、决策依据摘要、提交人及提交时间 | US-03 |
| AC-06 | CEO 已对某对象提交过一条决策记录（如 Go） | CEO 对同一周期同一对象再次提交新决策（如 No-Go） | 新决策记录独立写入审计日志，旧记录保持不变，[推导-待确认] 看板展示最新有效结论为 No-Go，历史记录仍完整可查 | US-01 |
| AC-07 | 非 CEO 角色用户已登录 | 该用户尝试对任意决策对象执行提交操作 | 系统拒绝操作，提示权限不足，决策标注提交入口不对非 CEO 角色开放 | US-02 |
| AC-08 | 系统在决策提交后因网络或其他原因通知推送失败 | 查询通知发送记录 | 通知状态标记为"发送失败"，[推导-待确认] 系统记录失败事件以便人工补发或自动重试；决策提交本身已成功，不因通知失败而回滚 | US-03 |
| AC-09 | CEO 已登录，当期已存在若干已提交的决策记录 | CEO 查看当期决策记录列表 | 列表完整呈现当期所有已提交决策（含对同一对象的多次决策），每条记录展示：决策对象、决策结论、决策依据、提交时间；所有记录均为只读 | US-04 |
