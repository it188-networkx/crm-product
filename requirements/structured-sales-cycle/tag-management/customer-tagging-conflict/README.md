# Feature Definition - [F-02] 客户打标与冲突解决 (Customer Tagging & Conflict Resolution)

> 需求特性定义（A0304）｜所属 Epic：[E-06] 客户标签体系管理 ｜ 所属主题：T-01 结构化销售周期
> **Feature 编号**：F-02 ｜ **优先级**：P0 ｜ **状态**：Draft

## 特性概述 (Feature Overview)

**所属 Epic**：[[E-06] 客户标签体系管理 (Customer Tag Management)](../README.md)

**业务定义**：
该特性负责将标签字典中已发布的标签值赋予客户档案，支持三种打标来源：系统按预定义规则自动匹配（自动打标）、AI 辅助建议（须人工确认后方可生效）与销售手动覆盖。当同一标签维度存在多个来源的赋值时，按确定性优先级规则解决冲突，确保同一客户在任意维度只保留一个有效标签值。所有打标操作及冲突解决结果均留存可检索的操作日志。

**范围边界**：

1. 包含：系统自动打标执行、AI 辅助建议的确认流程、销售手动覆盖打标、同维度冲突解决逻辑、打标操作日志留存。
2. 不包含：AI 建议的模型训练与评分逻辑（属 T-02/E-01 范围）、标签字典维护（见 F-01）、标签衰减执行（见 F-03）。
3. 参见 Epic [DEP-01]：本特性操作的标签值必须来自 F-01 已发布的标签字典；字典未就绪时自动打标和 AI 建议功能无法运行。
4. 参见 Epic [DEP-03]：AI 辅助打标建议的维度与字段格式须与 T-02 AI 特征工程需求对齐。

## 业务规则 (Business Rules)

- **[BR-01] 已发布标签约束**：自动打标与手动覆盖只能引用 F-01 已发布状态的标签条目；草稿或待审批状态的标签不得被赋值于客户档案。
- **[BR-02] AI 建议不自动生效**：AI 辅助打标建议须经销售或 MKT Leader 人工确认后方可写入客户档案；处于建议状态（待确认）的值不得参与冲突计算，也不得影响客户档案当前有效标签值（参见 Epic [BR-03]）。
- **[BR-03] 冲突解决优先级**：同一维度内多来源赋值冲突时，按以下优先级取唯一有效值：手动覆盖 > AI 建议已确认 > 自动打标（参见 Epic [BR-03]）；相同优先级来源之间取最新更新时间对应的值。
- **[BR-04] 手动覆盖持久化**：销售手动覆盖的标签具有最高优先级；后续自动打标不得覆盖同一维度已被手动覆盖的标签值，直至销售对该维度主动执行重置操作。
- **[BR-05] 冲突日志留存**：每次冲突解决须留存触发来源、参与冲突的全部值、解决结果与操作时间，支持按客户 ID 检索（参见 Epic [BR-04]）。
- **[BR-06] 打标日志留存**：所有打标操作（自动打标、AI 建议确认、AI 建议拒绝、手动覆盖、手动重置）须留存操作人（或系统标识）、操作时间、原值与新值，支持按客户 ID 和标签维度检索（参见 Epic [BR-04]）。
- **[BR-07] 手动覆盖权限**：销售执行层可对任意已发布维度执行手动覆盖；AI 建议的确认与拒绝须由销售或 MKT Leader 完成；系统自动打标不受角色约束。

## 业务对象定义 (Business Object Definition)

### 客户标签赋值 (Customer Tag Assignment)

客户标签赋值是客户档案在某一标签维度上当前有效的标签值记录，是打标来源竞争与冲突解决后的唯一业务结果。同一客户同一维度在任意时刻只允许存在一条有效赋值记录。

| 业务属性 | 业务说明 |
| :--- | :--- |
| 所属客户 | 标识该赋值归属哪一条客户档案 |
| 标签维度 | 标识赋值所属的标签分类维度（如行业、意向等级） |
| 标签值 | 当前有效的标签条目引用，须来自 F-01 已发布字典（参见 BR-01） |
| 赋值来源 | 标识本次有效值由哪种方式产生：自动打标 / AI 建议已确认 / 手动覆盖 |
| 赋值时间 | 本次有效赋值的业务生效时间，用于同级来源冲突时的时序判定（参见 BR-03） |
| 是否手动锁定 | 标识该维度是否处于手动覆盖保护状态，锁定状态下自动打标不得覆盖（参见 BR-04） |

### AI 打标建议 (AI Tagging Suggestion)

AI 打标建议是 AI 对某客户某标签维度生成的待确认推荐值，处于中间状态，不直接影响客户档案有效标签，需经人工操作后方可转化为有效赋值或被废弃。

| 业务属性 | 业务说明 |
| :--- | :--- |
| 所属客户 | 标识建议归属哪一条客户档案 |
| 标签维度 | 标识建议所属的标签分类维度 |
| 建议值 | AI 推荐的标签条目引用，须来自 F-01 已发布字典（参见 BR-01） |
| 建议置信指标 | AI 对本条建议的置信程度描述，用于辅助确认人判断是否接受建议（TBD：具体指标定义和阈值待 T-02 AI 特征工程对齐后确认；指标须对确认人可见以支撑决策；默认情况下所有建议均须人工判断，不因置信高低自动生效，参见 BR-02） |
| 建议状态 | 标识建议当前所处阶段：待确认 / 已确认 / 已拒绝 |
| 确认人 | 完成确认或拒绝操作的销售或 MKT Leader 标识（参见 BR-07） |
| 确认时间 | 人工操作发生的时间 |

### 标签变更日志 (Tag Change Log)

标签变更日志是单次打标操作或冲突解决的不可变操作记录，既承载打标行为的审计需求，也是冲突溯源的唯一数据来源。

| 业务属性 | 业务说明 |
| :--- | :--- |
| 所属客户 | 标识日志归属哪一条客户档案，支持按客户 ID 检索（参见 BR-06） |
| 标签维度 | 标识操作涉及的标签分类维度，支持按维度检索（参见 BR-06） |
| 操作类型 | 标识本次变更的触发方式：自动打标 / AI 建议确认 / AI 建议拒绝 / 手动覆盖 / 冲突解决 / 手动重置 |
| 操作人 | 人工操作的销售或 MKT Leader 标识；系统触发操作记录系统标识 |
| 原值 | 操作前客户档案在该维度的标签值；首次赋值时为空 |
| 新值 | 操作后客户档案在该维度的标签值；拒绝 AI 建议时新值与原值相同 |
| 操作时间 | 变更发生的业务时间 |

## 验收标准 (Acceptance Criteria)

| AC | 前置条件 | 触发动作 | 预期结果 | US Ref |
| :--- | :--- | :--- | :--- | :--- |
| AC-01 | 线索完成入池；目标维度存在已发布标签条目；自动打标规则与该线索匹配 | 系统执行自动打标 | 匹配到的已发布标签值写入客户档案，赋值来源标记为"自动打标"；留存打标操作日志（含系统标识、原值、新值、操作时间）（参见 BR-01、BR-06） | US-01 |
| AC-02 | 存在状态为"待确认"的 AI 打标建议；操作人角色为销售或 MKT Leader | 操作人确认 AI 建议 | 建议值写入客户档案，赋值来源标记为"AI 建议已确认"；建议状态更新为"已确认"；留存打标操作日志（参见 BR-02、BR-06） | US-02 |
| AC-03 | 存在状态为"待确认"的 AI 打标建议；操作人角色为销售或 MKT Leader | 操作人拒绝 AI 建议 | 建议状态更新为"已拒绝"；客户档案目标维度标签值不变；留存拒绝操作日志（参见 BR-02、BR-06） | US-02 |
| AC-04 | 目标维度存在已发布标签条目；操作人角色为销售执行层 | 销售提交手动覆盖，选择新标签值并填写原因 | 新标签值写入客户档案，赋值来源标记为"手动覆盖"，该维度进入手动锁定状态；留存手动覆盖操作日志（含操作人、原值、新值、操作时间）（参见 BR-03、BR-04、BR-06） | US-03 |
| AC-05 | 同一客户同一维度在同一时刻存在来自不同来源的赋值（如自动打标与已确认 AI 建议） | 系统执行冲突检测与解决 | 按 BR-03 优先级规则保留最高优先级来源的值作为有效标签；留存冲突日志（含触发来源、参与冲突的全部值、解决结果、操作时间）；同级来源冲突时取更新时间最新的值（参见 BR-03、BR-05） | US-04 |
| AC-06 | 目标维度已处于手动锁定状态 | 系统执行该维度的自动打标 | 自动打标结果不覆盖该维度当前有效标签值；锁定状态保持不变（参见 BR-04） | US-03 |
| AC-07 | 指定客户存在至少一条历史标签变更日志 | 按客户 ID 或标签维度发起日志查询 | 返回该客户完整的标签变更历史记录列表，包含每条日志的操作类型、操作人、原值、新值与操作时间（参见 BR-05、BR-06） | US-05 |
| AC-08 | 目标维度不存在已发布标签条目，或自动打标规则与该线索无匹配 | 系统尝试执行自动打标 | 该维度不产生赋值；客户档案目标维度状态保持为空；不产生日志条目（参见 BR-01） | US-01 |
