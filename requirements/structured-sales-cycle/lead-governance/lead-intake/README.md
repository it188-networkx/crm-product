# Feature Definition - [F-01] 线索录入与去重 (Lead Intake & Deduplication)

## 特性概述 (Feature Overview)

**所属 Epic**：[[E-01] 线索治理与分配 (Lead Governance & Assignment)](../README.md)

**业务定义**：
本特性用于将来自多个渠道的线索统一录入系统，并在录入过程中自动执行去重校验，确保线索池中同一客户不存在多条活跃记录。它关注的是"线索如何进入系统、重复记录如何识别与处置、异常来源线索如何标记"，不定义线索进入线索池后的评分、分配或培育行为。

**范围边界**：

1. 包含：多渠道线索录入（系统自动汇入与手工导入）、录入时去重校验与重复记录合并确认、来源异常线索的风险标记与操作人感知确认、客户类型标注。
2. 不包含：线索评分与分级（F-02）、线索分配路由（F-03）、线索池全局视图与批量操作（F-04）、评分规则配置（F-05）。

## 用户故事 (User Stories)

| ID | 角色 (As a...) | 想要 (I want to...) | 从而 (So that...) | 验收标准索引 (AC Ref) |
| :--- | :--- | :--- | :--- | :--- |
| US-01 | MKT Leader / 运维人员 | 将来自多个渠道的线索提交录入系统 | 建立统一可追踪的线索记录，消除线索遗漏 | AC-01, AC-05 |
| US-02 | MKT Leader | 在录入时查看系统识别的候选重复记录，并决定是否合并 | 确保同一客户在线索池中不出现多条活跃记录 | AC-02, AC-03 |
| US-03 | MKT Leader | 在录入时收到来源异常线索的风险提示并主动确认知晓 | 对可疑线索留存人工判断轨迹，避免劣质线索无感入池 | AC-04 |

## 业务规则 (Business Rules)

- **[BR-01] 多渠道录入支持**：线索录入须支持系统自动汇入（依赖 E-07 数据接入）与手工逐条录入两种方式；E-07 未就绪时降级为仅支持手工录入，线索来源渠道须在录入时明确标注（参见 Epic [DEP-01]）。
- **[BR-02] 去重校验必执行**：每条线索提交前系统必须自动执行去重校验（参见 Epic [BR-01]）；校验顺序：手机号精确匹配优先，企业名称精确匹配次之；客户主数据基线未就绪时降级为仅按手机号匹配（参见 Epic [ASM-01]）。
- **[BR-03] 重复记录处置**：去重校验命中时系统须向操作人展示候选重复记录，操作人须显式确认合并处置后方可完成提交；系统不得静默跳过或自动执行合并（参见 Epic [BR-01]）。
- **[BR-04] 合并语义**：合并操作以新录入线索的来源与联系信息字段为主记录；原记录中已存在的跟进历史与客户标签须完整保留至合并后记录；合并操作须在审计日志中记录主记录 ID、被合并记录 ID、操作人与操作时间（参见 Epic [NFR-03]）。
- **[BR-05] 异常来源标记**：来源渠道为空、或手机号与企业名称均缺失的线索，录入时须自动输出风险标签；风险标签不阻止录入完成，但操作人须在提交前显式确认已知晓该标记；客户类型为必填字段，提交时不允许为空值（参见 Epic [BR-08]）。

## 业务对象定义 (Business Object Definition)

### 线索记录

线索记录是客户意向信息进入系统后的基础承载实体，也是后续评分、分配与培育的操作对象。每条线索记录对应一名潜在客户，其唯一性由去重校验机制保障。

| 业务属性 | 业务说明 |
| :--- | :--- |
| 来源渠道 | 标识线索通过何种渠道进入系统（E-07 自动汇入或手工导入），是去重与风险判断的辅助依据 |
| 客户类型 | 标识线索对应客户的类型（终端客户 / 二次客户 / 外单客户），影响后续跟进节奏（参见 Epic [BR-08]） |
| 联系人姓名 | 线索对应的主联系人姓名，作为客户身份的辅助标识 |
| 手机号 | 作为去重校验的优先匹配字段，也是后续联系与追踪的核心标识 |
| 企业名称 | 作为去重校验的次级匹配字段，结合手机号共同确保线索唯一性 |
| 录入时间 | 记录线索正式进入线索池的时间戳，用于追踪来源时效与审计 |
| 去重状态 | 标识该线索在录入时的去重结果：唯一入池 / 合并确认入池 |
| 风险标签 | 标识线索是否在录入时被判定为来源异常，供后续人工复核参考 |
| 录入操作人 | 记录执行录入操作的人员标识，支撑操作审计 |

### 合并操作记录

合并操作记录是对两条线索合并行为的业务凭证，用于追溯合并决策的责任与时间，防止历史信息丢失。

| 业务属性 | 业务说明 |
| :--- | :--- |
| 主记录标识 | 合并后保留的线索记录标识，作为后续操作的引用锚点 |
| 被合并记录标识 | 合并前被废止的线索记录标识，用于历史追溯 |
| 操作人 | 确认执行合并操作的人员标识 |
| 操作时间 | 合并操作完成的时间戳 |

## 验收标准 (Acceptance Criteria)

| AC | 前置条件 | 触发动作 | 预期结果 | US Ref |
| :--- | :--- | :--- | :--- | :--- |
| AC-01 | 线索来源渠道已标注，手机号或企业名称已填写，客户类型已选择 | 提交录入新线索，去重校验无命中 | 线索以唯一入池状态进入线索池，录入全程耗时 ≤ 3 秒（参见 Epic [NFR-01]） | US-01 |
| AC-02 | 提交线索的手机号与线索池中已有记录精确匹配 | 提交录入操作 | 系统暂停提交，展示候选重复记录供对比；操作人确认合并后仅保留一条活跃记录，被合并记录的跟进历史完整保留，合并操作记录写入审计日志 | US-02 |
| AC-03 | 去重校验命中，系统已展示候选重复记录 | 操作人未完成合并确认即尝试完成提交 | 系统拒绝完成提交，不产生任何线索记录（参见 BR-03） | US-02 |
| AC-04 | 提交线索的来源渠道为空 | 提交录入操作 | 系统输出风险标签并要求操作人显式确认知晓；操作人确认后录入完成，线索携带风险标签进入线索池 | US-03 |
| AC-05 | 提交线索的客户类型字段为空 | 提交录入操作 | 系统阻止提交，提示客户类型为必填项（参见 Epic [BR-08]） | US-01 |
