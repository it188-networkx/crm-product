# Epic - [E-07] 数据接入与同步 (Data Integration & Sync)

## 概述 (Overview)

**业务背景：**

当前客户交互行为分散于邮件平台、企业微信、社媒渠道（LinkedIn/微博等）、官网表单、录音及线下日报等多个载体，数据孤岛导致线索来源信息无法自动汇入线索池；销售和市场人员需手工整理、录入互动记录，大量时间消耗在数据搬运而非有效触达。自动化记录缺失也导致 E-01 线索评分特征不完整，E-02 跟进记录无法覆盖自动触达的互动事件。

**核心目标**

本 Epic 旨在将主要外部渠道数据源统一接入线索池，实现客户交互行为的自动记录与映射，同时回写 CRM 主动外发邮件（EDM）产生的互动行为数据，为 E-01 提供完整线索来源与行为意向分，为 E-02 提供自动触达行为原始记录，为 E-06 提供行为特征数据。当自动接入不可用时，须提供可靠的手动导入作为容灾降级路径。

**价值预估：**

- **来源覆盖**：自动接入消除线索遗漏，为月均有效跟进客户数从 ≤ 30 家提升至 ≥ 100 家提供线索供给基础（支撑 T-01 KPI M6）。
- **效率解放**：行为数据自动映射取代手工录入，单次跟进备案工时从 ≥ 3 小时/次降低至 ≤ 30 分钟/次（支撑 T-01 KPI M6）。
- **数据完整性**：多渠道行为数据的完整汇入提升 E-01 评分特征丰富度，进而提升高意向线索识别准确率。

## 业务场景 (Business Scenarios)

### 场景一：新接入渠道配置

**用户**：管理员
**情境**：将企业邮件平台配置为线索行为数据来源，接入后邮件互动事件自动映射为线索跟进记录。
**业务流程**：

1. 管理员在接入管理中新增渠道配置，选择渠道类型（邮件平台），填写连接凭据（服务地址、认证信息）。
2. 系统进行连接测试，测试通过后渠道进入待激活状态；连接失败时展示错误诊断信息。
3. 管理员配置数据映射规则（邮件地址 → 客户主键，邮件事件类型 → 跟进记录动作类型；邮件渠道须区分普通往来邮件事件与主动 EDM 推送反馈事件，两类事件分别映射为不同动作类型的跟进记录）。
4. 管理员激活渠道，系统开始按配置的同步频率拉取增量互动数据。
5. 拉取的互动事件按映射规则转换为线索来源记录或 E-02 自动触达跟进记录写入（重复数据按去重规则处理）。

### 场景二：自动同步与异常处理

**用户**：系统（自动执行）/ 管理员（异常响应）
**情境**：日常运行中渠道同步偶尔遇到连接中断或数据格式异常，系统自动重试并告警。
**业务流程**：

1. 系统按配置周期尝试增量同步，遭遇连接失败时自动按有限次数重试（BR-05）。
2. 重试全部失败后，系统将该渠道标记为"同步异常"状态，向管理员推送告警通知。
3. 管理员查看异常日志，定位原因（连接超时 / 认证过期 / 数据格式变更等）并处理。
4. 问题处理后管理员手动触发补抓追赶同步，填补中断期间的数据缺口。
5. 补抓完成后渠道恢复"正常同步"状态，告警自动清除。

### 场景三：手动导入容灾

**用户**：MKT Leader / 销售执行层
**情境**：社媒渠道（如 LinkedIn、微博）暂时无法自动接入（社媒平台爱北爱南、爬虫维护成本高，手动导入是此类渠道的首选录入方式而非降级路径）， MKT 通过手动导入文件将渠道数据补录至线索池。除社媒外，其他尚无法自动接入的渠道同样可通过此方式完成数据录入。
**业务流程**：

1. 用户选择手动导入，上传标准格式数据文件（CSV/Excel）。
2. 系统解析文件，按配置的字段映射规则预览导入结果，标注可能冲突或缺失的字段。
3. 用户确认预览结果，系统执行导入并对重复记录按去重规则处理（提示合并）。
4. 导入完成后生成导入摘要（成功条数、跳过条数、异常条数），留存导入操作日志（承接 GC-04）。

### 场景四：EDM 外发行为回写

**用户**：系统（自动执行）/ MKT Leader（异常响应）
**情境**：CRM 主动对外发送的邮件（EDM）产生了打开、链接点击等互动行为，该行为数据需回写至对应线索的 `intent_score` 以完善行为意向分。
**业务流程**：

1. 邮件发送平台按配置的同步频率将成功发送、邮件打开、链接点击和订阅退出等互动事件回传系统。
2. 系统按字段映射规则将互动事件转换为线索行为事件记录，并更新对应线索的行为意向分属性，交由 E-01 评分引擎在下一次周期重新计算总分。
3. 异常时（如发送平台连接中断），系统按 BR-03 降级将该时段互动事件标记为待补录，提醒管理员手动补录。

## 业务规则 (Business Rules)

### 数据映射与去重

- **[BR-01]去重优先**：接入的外部数据在写入线索池前须先执行 E-01 定义的去重校验（手机号或企业名称匹配），重复数据按 E-01 BR-01 处理，不得绕过去重直接写入（依赖 E-01 去重规则）。
- **[BR-02]映射规则隔离**：不同渠道的数据映射规则相互独立，同一渠道的映射规则变更不影响其他渠道运行中的数据同步；映射规则变更须留存版本记录（承接 GC-07）。

### 同步可靠性

- **[BR-03]降级容灾**：自动接入渠道发生同步异常时，系统须在中断期间保留手动导入通道作为容灾路径，避免手动导入能力因自动接入就绪而被关闭。

### 同步重试

- **[BR-05]有限重试**：增量同步遭遇连接失败时，系统须按由短至长的间隔自动重试，且重试次数须有上限；全部重试失败后方可将渠道标记为异常状态并触发告警，不得无限重试。

### 数据安全

- **[BR-06]凭据访问控制**：渠道连接凭据的查看与修改权限须限制为管理员角色，普通用户不可见（承接 GC-06）；凭据须以加密方式持久化存储，不得以明文形式留存于任何持久化介质；凭据不得以明文形式出现于操作日志中。

## 非功能需求 (Non-Functional Requirements)

- **[NFR-01]同步时效**：增量同步的默认执行频率不超过 1 小时，确保外部互动事件在 1 小时内反映至线索池，支撑 E-01 评分特征的实时性。
- **[NFR-02]批量导入承载量**：手动导入须支持单次 ≥ 1000 条记录，导入完成时延不超过 30 秒，确保线下活动收集的大批量名片数据可快速录入。
- **[NFR-03]操作可审计**：渠道配置变更、手动导入操作、异常补抓须留存完整操作日志，支持按渠道、操作人、时间范围检索（承接 GC-04）。
- **[NFR-04]同步去重**：增量同步与补抓追赶过程中，同一外部渠道事件无论被触发多少次同步，系统均不得产生重复的线索或跟进记录；去重依据为外部渠道中该事件的来源唯一标识。

## 约束与依赖 (Constraints & Dependencies)

- **[DEP-01]E-01 去重规则依赖**：接入数据须服从 E-01 的去重校验规则，E-07 不独立维护去重逻辑；E-01 未就绪时，接入的数据需整体暂存，待 E-01 就绪后批量补处理。
- **[DEP-02]E-02 跟进记录写入依赖**：自动接入的行为数据（如邮件互动、社媒互动）须按 E-02 定义的跟进记录格式写入；E-02 未就绪时，行为数据仅作为原始事件暂存，不触发跟进记录创建。
- **[DEP-03]E-06 标签特征供给**：E-07 采集的行为特征（渠道来源、互动频次、内容类别）须能被 E-06 标签体系引用以驱动自动打标；E-06 未就绪时，行为特征仅在线索字段中存储，不自动转化为标签。
- **[DEP-04]外部渠道授权**：各渠道接入须获得对应平台的开放授权（如邮件平台的服务接入许可、企业微信开放平台授权）；授权未到位时对应渠道不可接入，降级为手动导入。
- **[ASM-01]渠道数据格式稳定**：假设已接入渠道的数据格式在接入后保持基本稳定，格式重大变更会触发映射规则失效告警；频繁格式变更的渠道数据质量风险由管理员承担。
- **[ASM-02]主动邮件发送能力待规划**：会议纪要（2026-03-04）将"邮件营销自动化（主动发送 + 反馈记录）"列为第一阶段高优先级方向；V1 中 E-07 仅覆盖邮件互动行为的接收侧记录，E-05 BR-05 已明确排除主动外发；主动发邮件能力在当前 Epic 集群中尚无 Epic 承接，需在后续需求评审中明确归属（新建 Epic 或纳入 E-05 后续迭代）。
- **[ASM-03]企业微信与社媒渠道自动接入技术可行性待定**：会议纪要（2026-03-04）将企业微信集成与媒体矩阵 API 集成列为"需技术调研"事项；调研结论未到位前，上述渠道的自动接入不应纳入 F-01 激活范围，须降级为 F-03 手动导入兜底；业务背景中提及上述渠道仅为范围描述，不代表接入方案已确认。
- **[ASM-04]官网渠道数据范围**：本 Epic 官网渠道接入范围限于可关联至具体线索的表单填写提交事件；官网浏览行为追踪（页面访问、导航路径等匿名事件）须先完成技术可行性评估，未确认前不纳入接入范围，对应数据降级为手动导入。
